# مشروع طبطبة - دليل المساهمة (CONTRIBUTING)

مرحباً بك في فريق تطوير "طبطبة". يهدف هذا الدليل إلى توضيح المعايير البرمجية واستراتيجية العمل لضمان جودة الكود وسهولة التسليم على مدار ١٢ أسبوعاً.

## 🌿 استراتيجية الفروع (Branching Strategy)

نتبع نظاماً خفيفاً للفروع لضمان الاستقرار:

- **main**: الفرع الرئيسي والمستقر (Production-ready). لا يتم التعديل عليه مباشرة.
- **develop**: الفرع الذي يتم دمج الميزات (Features) فيه بعد اختبارها.
- **feature/US-[ID]**: فرع لكل مهمة (User Story). مثال: `feature/US-02-05`.
- **fix/issue-name**: لإصلاح الأخطاء السريعة.

## 📝 مراجعة الكود (PR Checklist)

قبل فتح أي طلب دمج (Pull Request)، تأكد من الآتي:
- [ ] الكود يعمل بدون أخطاء برمجية (No Compilation Errors).
- [ ] تم تشغيل الـ Linter ولا توجد تحذيرات (No Lint Warnings).
- [ ] تم إضافة تعليقات توضيحية بالعربية للأجزاء المعقدة.
- [ ] تم اختبار الواجهة على أحجام شاشات مختلفة.

## 🛠️ أوامر الفحص (Lint Commands)

### Flutter
لتشغيل الفحص في مشروع Flutter:
```bash
flutter analyze
```

### NestJS (Backend)
لتشغيل الفحص في مشروع Backend (عند توفره):
```bash
npm run lint
```

## 🎨 المعايير البرمجية (Coding Standards)

- **اللغة**: الكود بالإنجليزية، والتعليقات والواجهة بالعربية.
- **التوثيق**: يجب إضافة تعليقات توضيحية مختصرة باللغة العربية لكل سطر برمجي في ملفات الواجهات (UI) والمنطق الأساسي (Core Logic) لضمان سهولة الفهم للفريق.
- **التسمية**: 
    - `CamelCase` للأصناف (Classes).
    - `snake_case` للملفات (Files).
    - `lowerCamelCase` للمتغيرات والدوال.
- **المجلدات**: تنظيم الشاشات في مجلدات حسب الدور (nurse, elderly, family, etc).
